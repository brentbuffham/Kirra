# Kirra UI Overhaul — Dockable Panel System

**Plan ID:** 20260307-1400-DockablePanelUI  
**Status:** PLANNED  
**Risk:** High (touches all UI layers, HTML structure, CSS, event handling)  
**Effort:** High (multi-phase, recommend incremental rollout)  
**Repo:** `brentbuffham/Kirra`  
**Branch suggestion:** `feature/dockable-panels`

---

## Goal

Replace Kirra's current side-nav + floating toolbar panel layout with a **dockable panel system** where every panel (Explorer/TreeView, Properties, Timeline, Tools, Data Explorer, Charging, Statistics) can be:

- **Docked** into configurable positions (left, right, bottom, center tabs)
- **Resized** via draggable splitters between panels
- **Popped out** to separate browser windows (for multi-monitor workflows)
- **Collapsed** to a vertical tab strip on the right edge when popped out
- **Restored** by clicking the strip tab or closing the popup window
- **Tab-grouped** so multiple panels share a dock zone with tab switching

All panels must use the existing **FloatingDialog visual style** (4px border-radius, `--dark-mode-border` borders, `--primary-color` headers, `--mouseover-color` red hover states). No new visual language — the docked panels should look like FloatingDialogs that happen to be embedded in a layout grid.

---

## Key Constraints (from CLAUDE.md)

1. **Vanilla JavaScript only** — no React, Vue, or framework. Kirra uses Vite + vanilla JS with ES modules.
2. **FloatingDialog is the base class** — "All Dialogs, Popups, Menus should be constructed from FloatingDialog.js" (CLAUDE.md line 593-601). The dock panel wrapper should extend or compose FloatingDialog, not replace it.
3. **No template literals** — Kirra convention is string concatenation.
4. **Global class exposure** — classes are assigned to `window.*` for cross-module access.
5. **Minimize draw calls** — panel resize events must debounce Three.js renderer resize.
6. **Z-up coordinate system** — canvas/Three.js orientation is unchanged by this work.
7. **kirra.js monolith reduction** — any new UI code goes in `src/dialog/` or a new `src/layout/` module folder, NOT in kirra.js.
8. **CSS variables** — use existing `:root` vars from `kirra.css`, do not introduce new colour values.

---

## Architecture Decision: dockview-core vs Custom

**Recommendation: Use `dockview-core` (npm package)**

- Zero dependencies, vanilla TypeScript, ~54k weekly downloads
- `npm install dockview-core` — provides `DockviewComponent` class
- Panel lifecycle via `createComponent(options)` callback — returns `{ init(params) { ... } }`
- Built-in: tab groups, splitters, drag-to-dock, popout windows, layout serialization
- Kirra wraps each panel's content DOM inside dockview's container elements
- Styling override: apply Kirra's CSS variables to dockview's CSS classes

**Alternative: Fully custom (no library)**

- Build a `DockLayout` class from scratch using flexbox + splitter divs
- More control, no dependency, but significant effort for popout/serialization
- Only choose this if dockview-core proves incompatible with Kirra's vanilla DOM approach

The plan below is written to work with **either approach** — the panel content modules are the same regardless.

### 0.3 — Create DockPanel wrapper class (extends FloatingDialog pattern)

**File:** `src/layout/DockPanel.js`

This is the visual wrapper for every docked panel. It must look identical to FloatingDialog but support docked vs floating vs popped-out states.

```
DockPanel
├── header (floating-dialog-header style: 4px radius, --primary-color bg, drag cursor)
│   ├── title text
│   ├── tab bar (when multiple panels share a dock zone)
│   └── action buttons: [popout ⧉] [maximize □] [close ✕]
├── body (floating-dialog-content style: 8px padding, overflow-y auto)
│   └── PanelContent module mounts here
└── resize handle (when not maximized)
```

**CSS class mapping (reuse existing Kirra classes):**

| DockPanel element | Kirra CSS class | Source |
|---|---|---|
| Outer container | `.floating-dialog` | kirra.css:2795 |
| Header bar | `.floating-dialog-header` | kirra.css:2824 |
| Body content | `.floating-dialog-content` | kirra.css:2847 |
| Tab buttons | `.dock-tab` (new, styled like `.tree-panel-btn`) | kirra.css:2021 |
| Action buttons | `.tree-panel-btn` | kirra.css:2021 |
| Close button hover | `--mouseover-color` (#ff0000) | kirra.css:35 |
| Confirm/active | `#00a91c` (green) | kirra.css:2892 |

**Key difference from FloatingDialog:** DockPanel does NOT use `position: fixed`. When docked, it uses `position: relative` and participates in flexbox layout. When floating/popped, it switches to `position: fixed`.

## Phase 1: Layout Shell (Breaking Change — Behind Feature Flag)

### 1.1 — Create DockLayout manager

**File:** `src/layout/DockLayout.js`

The central layout manager that owns the dock grid.

```
DockLayout
├── menubar (existing, unchanged)
├── dock-area (new flexbox container, replaces .main-content)
│   ├── left-dock (vertical strip — tool panels)
│   ├── splitter-h
│   ├── center-column
│   │   ├── center-dock (3D viewport + tabs)
│   │   ├── splitter-v
│   │   └── bottom-dock (timeline, table, stats)
│   ├── splitter-h
│   ├── right-column
│   │   ├── right-dock-top (explorer/tree)
│   │   ├── splitter-v
│   │   └── right-dock-bottom (properties/charging)
│   └── popout-strip (right edge, 0px width until panels popped)
└── statusbar (new, replaces scattered status elements)
```

**Responsibilities:**
- Create the flexbox grid structure
- Manage splitter drag-to-resize (horizontal and vertical)
- Track which DockPanels are in which dock zones
- Handle popout: hide panel → add to right-edge strip → open `window.open()`
- Handle dock-back: close popup → remove from strip → show panel → restore size
- Serialize/deserialize layout to localStorage (optional, Phase 3)
- Expose `window.dockLayout` for cross-module access

### 1.2 — Splitter implementation

**File:** `src/layout/Splitter.js`

Simple draggable dividers between dock zones.

```javascript
class Splitter {
    constructor(direction, leftOrTopElement, rightOrBottomElement) { ... }
    // mousedown → track drag → update flex sizes → mouseup
}
```

**Styling:**
- 4px wide/tall (matches existing `--canvas-border` pattern)
- Background: `var(--dark-mode-border)` (#4d4d4d)
- Hover/active: `var(--mouseover-color)` (#ff0000)
- Cursor: `col-resize` or `row-resize`

### 1.3 — Popout strip (right edge dock)

**File:** `src/layout/PopoutStrip.js`

A thin vertical container on the far right edge of the dock area.

- Initially 0px width, invisible
- When a panel is popped out: expands to 26px, shows a vertical tab button
- Tab button has: pulsing red dot + panel name (rotated text, `writing-mode: vertical-rl`)
- Click tab → calls `dockLayout.dockPanel(panelId)` to restore
- Styled with existing `.floating-dialog` border/bg colours

### 1.4 — Feature flag and gradual migration

Add to kirra.js initialisation:
```javascript
// Feature flag: set to true to enable new dockable layout
window.useDockableLayout = false;
```

When `true`:
- Hide the old side-navs, toolbar panels, and tree panel
- Mount DockLayout into `.main-content`
- Mount panel content modules into their default dock positions

When `false`:
- Existing UI unchanged — no regression risk

### 1.5 — Viewport panel integration

The 3D viewport (canvas + Three.js) is the most critical panel. It must:

- Mount the existing `canvas2D` and Three.js renderer into a DockPanel body
- Listen for panel resize events → call `renderer.setSize(w, h)` and update camera aspect
- Debounce resize to avoid hammering WebGL (use `requestAnimationFrame`)
- When popped out to a new window: **create a new WebGL context** in that window (browser limitation — WebGL contexts can't move between windows)
- When docked back: destroy the popup renderer, re-attach to main window renderer

**This is the hardest part of the entire plan.** The viewport popout needs careful handling.

### 3.4 — Context menu integration
- Context menus (`ContextMenuManager.js`) continue to work unchanged
- They spawn on top of the dock layout at click position
- No changes needed if z-index hierarchy is maintained (context menu z-index > dock panels)

### 3.5 — FloatingDialog coexistence
- FloatingDialogs continue to spawn on top of everything (z-index 10000+)
- They are NOT docked — they remain floating modal/modeless dialogs
- The dock system handles persistent panels; FloatingDialogs handle transient forms/confirmations
- `openDialogsSet` and `openDialogsByTitle` tracking (FloatingDialog.js lines 10-12) remain unchanged

## Risk Mitigation

1. **Feature flag** — entire system behind `window.useDockableLayout`, old UI preserved
2. **Incremental panel migration** — one panel at a time, each is a separate testable PR
3. **No kirra.js modifications in Phase 0** — only new files added
4. **Viewport popout deferred** — pop out simpler panels first (Explorer, Properties), viewport popout is Phase 2+ when the pattern is proven
5. **CSS additions only** — new classes added, existing classes modified minimally, never deleted until old UI code is removed
